原版本的 6 个 "Toy 味" 问题

03 / 11
  1. ① 同手机号可无限刷验证码（无 60s 间隔，无日上限）—— 一夜烧光短信费
  2. ② 验证码明文存 Redis —— Redis 一旦泄露 = 验证码全裸
  3. ③ 验证码错误次数不限 —— 6 位数字暴力枚举可破
  4. ④ 短信发送写死成 log.debug —— 永远接不上真实服务商
  5. ⑤ Token 续期每次请求都写 Redis —— 写放大，热 key 风险
  6. ⑥ 用户信息用 Hash 存，字段全转 String —— 类型丢失，DTO 还得手动转
这 6 个点，企业里任何一个都会被 review 打回。

安全层 ①：防刷 + 哈希 + 锁定

06 / 11

三道闸门

  • 冷却：60s 内重复发同号 → 拒绝
  • 日上限：单号每日 10 次 → 拒绝
  • 锁定：连错 5 次 → 锁 10 分钟，期间发码/登录都拒

哈希存储

Redis 存 SHA-256(phone + code + salt)，盐通过 hmdp.login.code-salt 配置。

就算 Redis 被脱库，攻击者也拿不到原始 code。

stateDiagram-v2
  [*] --> Idle
  Idle --> Cooldown: 发码
  Cooldown --> Idle: 60s 后
  Idle --> Verifying: 用户提交
  Verifying --> Idle: 成功(清失败计数)
  Verifying --> Failed: 失败 fail+1
  Failed --> Idle: fail < 5
  Failed --> Locked: fail ≥ 5
  Locked --> Idle: 10min 后自动解锁

安全层 ②：SmsClient 抽象

07 / 11

为什么要抽象

  • 原版直接 log.debug，永远上不了生产
  • 不同环境需要不同实现：日志 / 阿里云 / 腾讯云 / 自研网关
  • 测试态可注入 Mock，记录调用

实现替换方式

新增 AliyunSmsClient implements SmsClient，加 @Primary 或 @ConditionalOnProperty 即可，UserServiceImpl 零改动。

会话改造 ②：滑窗续期

09 / 11

原版问题

每次请求 → EXPIRE 一次。活跃用户每秒 N 个请求 = 每秒 N 次 Redis 写。

改造

读 token 后先 TTL，仅当剩余 < 阈值（默认 600s）才 EXPIRE。

Long ttl = redis.getExpire(key, SECONDS);
if (ttl > 0 && ttl < props.getRefreshThresholdSeconds()) {
    redis.expire(key, props.getTokenTtlSeconds(), SECONDS);
}

双拦截器设计

10 / 11 
flowchart TB
  R[请求进入] --> I1{RefreshTokenInterceptor
order=0, 拦截 /**}
  I1 -->|有 token & 有效| W[ThreadLocal 存 user
必要时滑窗续期]
  I1 -->|无 token / 无效| P1[直接放行]
  W --> I2{LoginInterceptor
order=1, 排除公开路径}
  P1 --> I2
  I2 -->|公开路径| H[Handler]
  I2 -->|受保护路径 & ThreadLocal 为空| X401[401]
  I2 -->|受保护路径 & 已登录| H
  H --> AC[afterCompletion: 清理 ThreadLocal]
为什么拆两个？因为续期需要对所有请求生效（包括 /shop 这种不需要登录的查询，登录用户访问也希望续期），但拦登录态只能限定在受保护路径。一个拦截器同时承担两件事 → 排除规则会互相打架。
